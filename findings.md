## 实施环境与基础

- 2026-07-13 开始实施时仓库位于干净的 `main`，已有设计与实施计划提交。
- 本机工具链为 Go `1.26.0`（`darwin/arm64`）和 Node.js `24.13.0`，满足 Task 1 的版本要求。
- Task 1 使用 Chi v5 提供最小路由，健康端点仅返回进程存活状态；存储就绪检查留给 Task 3。
- 本机 TMDB 凭据文件名预留为被 Git 忽略的 `.tmdb-token`，在 Task 6 需要真实上游验证前不创建，也不进入命令、日志或提交。
- Task 2 配置直接读取进程环境，不加载 `.env`；生产环境默认 Secure Cookie，开发环境默认关闭，并允许显式布尔覆盖。
- `APP_ENCRYPTION_KEY` 当前契约为 Base64 编码的 32 字节值；格式错误只返回稳定错误，不包含原始输入。
- 请求日志只记录 URL path，不记录 query、header 或 body；`slog` handler 额外按敏感属性名和已知秘密值统一脱敏。
- panic 恢复不记录 panic 值或堆栈，只返回带稳定 `internal_error` 与 `requestId` 的 RFC 9457 Problem Details。
- Task 3 使用 modernc SQLite `v1.53.0`，保持 `CGO_ENABLED=0` 路径；写池固定 1 个连接，读池固定 4 个并启用 `query_only`。
- SQLite 连接统一启用 `foreign_keys`、WAL 和 5 秒 `busy_timeout`；应用只以 `0700` 创建缺失的数据目录，不修改已有挂载目录权限。
- 嵌入迁移按版本排序、单项事务执行，并记录名称与 SHA-256；已应用版本的内容或名称发生变化时启动被拒绝。
- `/healthz` 只表示进程存活；`/readyz` 只有在数据库已迁移且写事务可建立时通过，内部 SQLite 错误不会暴露给客户端。
- Task 4 的密码格式固定为 Argon2id `m=65536,t=3,p=2`、16 字节 salt、32 字节派生值；验证器严格解析并对畸形值安全失败。
- 会话与 CSRF 使用独立随机值，数据库只保存各自 SHA-256；重新登录原子撤销该用户的旧会话，默认会话期限 30 天。
- 登录失败桶使用规范化用户名与客户端 IP 的 SHA-256 键；15 分钟窗口内 5 次失败后阻断 15 分钟，不在数据库保存原始组合。
- 会话 last-seen 最多每 5 分钟写入一次；过期、撤销、停用用户和未知令牌统一表现为无效会话。
- setup/login/logout 的非 GET 请求严格校验同源 Origin；logout 还使用与会话绑定的 CSRF 哈希，Cookie 为不透明、HttpOnly、SameSite=Lax 并按生产配置启用 Secure。
- Task 5 固定 React/React DOM `19.2.7`、Vite `7.3.6`、TypeScript `7.0.2` 和 Vitest `4.1.10`；plugin-react 使用与 Vite 7 兼容的 `5.2.0`。
- 前端令牌严格采用设计文档的 OKLCH 中性色、主色与强调色，固定字号/间距/圆角/z-index，并支持系统暗色、显式主题和 reduced-motion。
- 应用壳在 `1440x900`、`768x1024`、`375x812` 实测均无横向溢出；侧栏分别为 216/72/0px，移动底栏 65px，内容预留 88px 底部空间。
- 浏览器 DOM 检查确认导航、搜索、标题与分区语义完整；移动搜索按钮能把焦点送到全局搜索框，控制台无警告或错误。
- Inter 字体文件尚未加入：设计要求先提交许可证，因此当前只声明字体栈并使用系统回退，后续取得可提交许可证时再自托管资产。
- Task 6 的 TMDB 客户端默认 8 秒超时，搜索缓存 6 小时，电影/剧集/季/集详情缓存 7 天；缓存命中发生在令牌检查之前，已缓存数据可在未配置或上游不可用时读取。
- 缓存键为请求路径与排序参数的 SHA-256，数据库只保存重新编码后的类型化 DTO；未知上游字段被剥离，不能把意外回显的令牌带入数据库或备份。
- HTTP 层将 TMDB snake_case 隔离在适配器内，对前端只输出 camelCase；限流、超时、未配置和不可用分别映射稳定错误码，不透传上游正文。
- TMDB 设置状态只读且不提供令牌输入；当前 Settings 路由始终显示官方英文归属声明和 `https://www.themoviedb.org/` 外链。

- Task 6 无需真实上游数据即可完成合约验证，因此未创建 `.tmdb-token`，未在命令、日志、截图、缓存或提交中使用个人令牌。
- Task 7 将本地 UUID 保存在 `media_items`，外部身份独立保存在 `media_external_ids`，唯一键固定为 `(source, source_id, media_type)`；业务外键不使用 TMDB ID。
- `media_items` 将外部快照与可空自定义标题/简介分列保存；刷新只更新外部列，有自定义值时读取始终优先返回用户字段。

- 评分和笔记区分“请求中省略”与“显式 null”：省略时保留原值，显式 null 时清空并递增版本。

## `invoice-manage` 工程参照

- 前端为 React 19 + TypeScript + Vite，后端为 Python 3.12 + FastAPI + SQLAlchemy。
- Dockerfile 使用 Node 构建前端、Python slim 运行后端，并把静态产物复制进后端形成单应用镜像。
- 容器使用非 root 用户，暴露固定的 `8080` 端口；Compose 配置 `no-new-privileges`、丢弃 capabilities、健康检查和命名卷。
- Compose 额外编排 PostgreSQL、Redis、后台 worker 与持久化卷；应用镜像明确面向 `linux/amd64` 和 `linux/arm64`。
- 这套模式适合需要异步作业的业务。video-record 是否需要独立 worker/Redis，应由元数据同步、批量导入和媒体服务器同步的复杂度决定，不能仅为照搬参照项目而引入。

## 竞品调研

### MoviePilot v2（第一轮）

- 官方前端仓库 `jxxghp/MoviePilot-Frontend` 的 `v2` 分支显示其基于 Vue 3、Vuetify 3、Vite，支持中英文和远程插件组件；这说明它的 UI 首要服务于可扩展的 NAS 自动化控制台，而非个人观影日志。
- 公开搜索结果与教程截图反复出现仪表盘、媒体搜索、订阅、整理、下载器/媒体服务器配置等控制台场景，信息密度较高，桌面端侧边导航明显。
- 可借鉴：海报主导的影视浏览、全局搜索入口、桌面侧栏的信息分组、任务状态的即时反馈、暗色环境下清楚的层级。
- 不应照搬：下载/订阅/整理工作流、插件中心、运维仪表盘、复杂系统设置，以及把海量配置暴露给日常记录用户。
- 证据来源：MoviePilot 官方 GitHub 仓库、官方站点与公开教程搜索结果；后续继续核对具体路由、导航与详情结构。

### Jellyseerr / Seerr（第一轮）

- Jellyseerr 与 Overseerr 已合并为 Seerr。官方定位仍是“媒体发现 + 请求管理”，支持 Jellyfin、Plex、Emby，并展示主面板、内容筛选、请求和趋势等界面截图。
- 首页信息架构以海报和横向内容轨道为主，详情页强调评分、演职员、流媒体可用性等易消化信息；电影与剧集共享发现和搜索入口，剧集在详情内进入季度选择。
- 官方明确强调移动端体验、25+ 语言、SQLite/PostgreSQL 双数据库、观看清单和细粒度权限，说明其自托管产品同时照顾轻部署与多人使用。
- 可借鉴：海报优先的内容发现、清晰的电影/剧集详情层级、统一搜索、响应式移动体验、内容状态覆盖在海报或主操作附近。
- 需要转译：Seerr 的单一主动作是“请求”，本项目应替换为“记录/更新状态”，并把想看、在看、看过、观看日期、评分和笔记作为个人域数据。
- 不应照搬：季度请求、审批/拒绝、请求限额、Radarr/Sonarr、通知代理、媒体可用性管理和复杂权限矩阵。
- 证据来源：Seerr 官方站点、官方文档入口与公开搜索结果。

### 已阅（第一轮）

- 官方 App Store 将其定义为简洁无广告的书籍、电影、电视剧记录工具；核心状态为已看、在看、想看。
- 核心记录闭环是一键搜索添加，再记录评分、日期和评价；剧集与阅读均支持进度追踪，日历可显示每日在看进度并按完成状态筛选。
- 长期价值来自月度/年度总结、日历回顾、二刷三刷、标签、书单/影单和海报分享，而不是内容消费或社区互动。
- 数据策略强调本地保存、离线可用，并提供 iCloud、网盘、Excel、导入导出；也允许自定义录入搜索不到的条目。
- 用户评论反复强调“界面简洁、没有花哨功能、每天一分钟完成记录”，说明低操作成本与私人角落感是其主要竞争力。
- 可借鉴：三态记录、快速添加、单日剧集进度、重复观看、日历回顾、年度总结、自定义条目、数据导入导出。
- 需要调整：本项目只做影视，不复制书籍域；家庭多用户和服务器同步不能破坏个人数据归属与快速手动闭环。
- 证据来源：中国区 App Store 官方产品页（应用 ID `1589203887`）及其公开评论/开发者回复。

### Apollo（第一轮）

- 官方定位是记录看过的电影/剧集，并管理追剧进度与待看清单；提供高清海报、剧照、演职员和分集资料。
- 单次观影记录比“已阅”更细：评分、评论、日期之外，还可选择观看方式/平台、存档海报样式、自定义标签，并支持二刷和弃看。
- 片单、统计图表、发现、年度观影报告和分享海报共同构成长期回顾与展示能力；支持导入豆瓣数据和 iCloud 同步。
- 用户反馈指出长篇剧集只按“第几季第几集”并不足够，还需要可感知的总集序号；这应进入本项目的剧集进度设计。
- 可借鉴：观看方式、弃看、二刷、标签维度统计、豆瓣等历史数据导入、剧集进度双表达。
- 不宜首版照搬：海报样式选择、重发现推荐和分享能力可后置，避免增加快速记录的表单负担。
- 证据来源：中国区 App Store 官方产品页（应用 ID `1616467801`）及公开评论。

### Filmo（初步）

- 官方商店索引将其定义为私人书影音收藏，支持记录进度与感受、标记与谁共同体验、多次重温、个性化推荐与文化收藏。
- 对本项目最有价值的差异点是“共同观看者”：家庭成员可以共同参与某次观看，但每个人的个人评分、笔记和状态仍应独立。
- 当前公开评分样本很少，且美国区详情在当前地区被重定向，不能把它作为成熟度或具体界面细节的强证据。
- 可借鉴：一次观看事件可关联多名家庭成员；产品语气强调私人收藏与共同回忆。
- 不应照搬：书籍、音乐域和个性化推荐不进入首版。
- 证据来源：中国区/美国区 App Store 精确应用 ID `6741805793` 的索引摘要与公开搜索结果。

### 横向结论

- MoviePilot v2 提供“自托管控制台”的桌面导航和状态表达，但其业务密度过高。
- Jellyseerr/Seerr 提供“影视内容发现与详情”的海报组织方式，但请求域必须彻底替换为个人记录域。
- 已阅提供最低成本、最完整的日常记录闭环和日历/年度回顾。
- Apollo 提供更专业的观影字段、剧集进度、片单与统计维度。
- Filmo 提供家庭场景中的共同观看语义。
- 本项目的组合方向应是：Seerr 的影视浏览清晰度 + 已阅的记录效率 + Apollo 的影视专业度 + Filmo 的家庭记忆；明确排除 MoviePilot/Seerr 的资源自动化与请求管理。

## 已确认产品方向

- 用户模型采用“家庭多用户架构、单人默认体验”：每位成员拥有独立观影记录，后续可支持家庭内共享；不以公开社区为首要目标。
- `invoice-manage` 只证明 Docker 与多架构发布模式可行，不构成 React、FastAPI、PostgreSQL、Redis 或 worker 的选型依据。
- 技术选型必须在产品流程和外部同步边界明确后独立比较，优先考虑自托管维护成本、ARM64/AMD64 可构建性、镜像体积、数据迁移与备份难度。
- 手动搜索、标记、评分和笔记构成无需媒体服务器也能使用的完整体验。
- Jellyfin、Emby、Plex 是可选播放历史来源；同步事件应转换为幂等的观看记录候选，不能无条件覆盖人工编辑。
- TMDB 是影视搜索和基础元数据的上游。系统需缓存影视身份与必要快照，将 `tmdb_id + media_type` 作为外部身份，并把用户评分、观看日期、状态、笔记等数据保存在独立的本地域模型中。
- TMDB 不可用或限流时，已缓存条目和既有个人记录仍应可读、可编辑；新增搜索需要明确错误与重试机制。
- 首版产品重心为观影档案：搜索、想看、在看/看过、观看日期、评分、笔记、重复观看、剧集进度与回顾统计。
- 明确排除求片、下载器、订阅、媒体文件管理和公开社区；不复制 Jellyseerr 的请求域，也不复制 MoviePilot 的自动化媒体管理域。
- 视觉方向已确认：克制、私人、具有电影质感；默认跟随系统明暗主题，由海报承担主要色彩，中性界面只使用少量状态色。
- 无障碍默认采用 WCAG 2.2 AA，并要求键盘可用、可见焦点、减弱动态效果和不依赖颜色表达状态。

## 产品注册类型

- 明确属于 `product`：设计服务于搜索、标记、回顾、筛选和统计等持续任务，不是营销展示站。

## 工程方案比较

### 方案 A：Go 模块化单体 + React/Vite + SQLite（推荐）

- Go 进程提供 API、认证、TMDB/媒体服务器集成、轻量定时同步，并直接托管 Vite 静态产物；部署时只有一个应用容器和一个数据卷。
- SQLite 使用 WAL、外键和迁移，适合个人/家庭的并发规模；备份的核心是一个数据库文件，必要时再加本地缓存目录。
- 前端选 React 不是继承 `invoice-manage`，而是基于成熟的服务端状态缓存、表单、无障碍组件和测试生态；后端、存储、任务与运行时完全独立选型。
- 优点：镜像和内存占用低，ARM64/AMD64 构建直接，升级/备份简单，不需要 PostgreSQL、Redis 或独立 worker。
- 代价：需要明确单实例写入约束；若未来发展为公网大规模服务，再迁移 PostgreSQL 和独立任务队列。

### 方案 B：SvelteKit 全栈 TypeScript + SQLite

- 页面、API、认证和数据库访问统一在 Node 进程中，界面开发速度快，代码语言单一。
- 优点：项目结构紧凑，SSR/PWA 能力自然，首版开发效率高。
- 代价：后台同步、长任务和进程内调度需要额外约束；运行时镜像和依赖面通常大于 Go 单体，长期全栈耦合更强。

### 方案 C：前后端分离 + PostgreSQL + Redis/worker

- 前端 SPA、API、数据库和同步 worker 分开部署，天然支持多实例与大规模异步作业。
- 优点：扩容和复杂任务能力最强。
- 代价：对个人/家庭自托管明显过度，安装、备份、故障定位和多架构发布成本最高；与“低运维私人影库”目标冲突。

### 初步推荐

- 选择方案 A，并明确单实例运行；采用持久化任务/同步运行记录，进程重启后补跑到期任务。
- 首版不引入 Redis、消息队列或独立 worker；只有实际性能数据证明需要时才拆分。
- 数据库层保留 repository 边界和迁移测试，避免把 SQLite 特有 SQL 扩散到整个领域层，为未来迁移保留现实路径。

### 已确认技术方向

- 用户确认采用方案 A：Go 模块化单体 + React/Vite + SQLite，单应用容器、单实例运行。
- TMDB 凭据只允许使用服务端环境变量 `TMDB_READ_ACCESS_TOKEN`；不得写入 `.env.example` 的真实值、源码、测试夹具、前端变量、Dockerfile、Compose 默认值、日志、错误响应、导出包或数据库备份。
- 管理界面只能显示凭据状态和可选指纹尾部，不回显完整令牌；日志中 Authorization 头必须统一脱敏。
- 设计阶段不需要调用实际 TMDB API；凭据不在仓库中持久化。

## 已确认设计章节

### 第一章：信息架构与页面边界

- 一级导航固定为：首页、影库、日历、统计、设置。
- 全局搜索是跨页能力，不作为一级导航；同时返回 TMDB 与本地缓存结果，并显示媒体类型、年份和个人状态。
- 影视详情以“记录/更新记录”为主操作，个人状态与观看历史优先于演职员等外部资料。
- 首页最多保留“继续观看”和“最近记录”两个横向区域，避免流媒体式无限内容轨道。
- 桌面采用窄侧栏 + 顶部搜索；平板折叠侧栏；移动采用底部导航和安全区上方固定主操作。
- 重复观看新增事件而非覆盖旧记录；共同观看关联成员但不代替成员填写个人评分或笔记。
- 同步结果只有在身份匹配明确且无冲突时才可自动落档，否则进入核对流程。

## 当前设计提案

### 第二章：领域数据模型（已确认）

- `media_items`：本地影视身份与 TMDB 快照，使用本地 UUID；外部唯一约束为 `source + source_id + media_type`。电影与剧集共享基础表，季、集使用独立表。
- `user_media_states`：每位成员对影视的当前聚合状态，仅允许 `none/wishlist/watching/completed/dropped`；包含个人评分、私密笔记、开始/完成日期。
- `watch_events`：不可被状态替代的事实记录，保存开始/结束时间、电影或具体集、观看方式、来源、完成度和可选备注；重复观看自然表现为多条事件。
- `watch_event_participants`：一次事件关联一名或多名家庭成员；事件事实共享，但每人的状态、评分和笔记仍在各自个人域。
- `episode_progress`：每位成员对单集的已看状态和最近进度；系列进度由单集数据计算，同时呈现 `S02E03` 与全剧累计集数。
- `tags` / `user_media_tags`：标签为用户私有，首版不做家庭全局标签词库。
- `collections` / `collection_items`：个人片单，首版支持手动排序，不承担社区分享。
- `external_accounts`、`external_media_mappings`、`sync_runs`、`sync_candidates`：保存媒体服务器配置引用、外部 ID 映射、同步游标/结果和待核对候选；服务端密钥加密存储或使用环境变量，不返回明文。
- `audit_events`：记录登录、成员管理、集成配置、导入、恢复和批量删除等高风险动作，不记录个人笔记正文或凭据。
- 评分数据库统一存为 `0–100` 整数，界面默认显示 `0.0–10.0`，便于导入五星制、十分制或百分制数据而不丢精度。
- 来源优先级：用户手动值 > 已确认导入值 > 已确认同步值 > 外部元数据默认值；自动同步不得覆盖更高优先级字段。

### 第三章：核心交互与异常处理（已确认）

- 首次启动进入封闭初始化页，创建首位管理员并检测数据目录、数据库迁移与 TMDB 配置状态；成功后关闭公开初始化入口。家庭成员只能由管理员创建或重置密码，不开放公网注册。
- 搜索采用 300 ms 防抖；先显示本地匹配，再合并 TMDB 结果并去重。条目卡显示电影/剧集、年份、原名和个人状态。
- 快速记录采用“快速层 + 完整层”：默认只要求状态；完成状态默认创建今天的观看事件。评分、日期、观看方式、共同观看者、标签和笔记可展开补充。
- 电影记录：`想看` 只更新状态；`看过` 新增事件；`重看` 永远新增事件；删除事件后重新计算当前状态和首/末观看日期，但不删除个人评分、笔记或标签。
- 剧集记录：支持单集勾选、连续多集批量标记、季全选；默认日期为今天。下一集入口只推进一集，操作后可撤销。状态根据进度建议变化，但用户选择的 `dropped` 不被自动改回。
- 同步候选分为精确匹配、可能匹配、无法匹配和冲突。精确匹配且无更高优先级数据时自动落档；其他情况必须人工确认、重新匹配、忽略或创建自定义条目。
- TMDB 搜索失败时不清空查询与本地结果，显示可重试错误；已缓存详情继续可用。限流遵循上游重试时间并使用退避，不向前端泄漏上游响应或凭据。
- 所有保存使用幂等请求键和乐观并发版本；重复提交不产生重复事件。版本冲突保留用户当前输入，显示服务器最新值并提供重新应用。
- 删除观看事件、清空个人记录、删除成员、断开集成和恢复备份需要确认。单条普通字段保存、状态修改和进度更新使用可撤销提示，不弹确认框。
- 表单错误就地关联字段并把焦点移到第一处错误；网络错误保留输入。后台同步使用非阻塞状态提示，不用全屏加载。
- 自定义条目仅在 TMDB 无结果或同步无法匹配时出现，最少要求标题、媒体类型和年份可选；之后允许管理员手工关联 TMDB，合并时保留全部个人记录。

### 第四章：视觉系统与响应式规范（已确认）

- 使用场景句：用户在夜间看完一集后用手机快速记录，也会在白天用桌面浏览多年档案；因此主题跟随系统，暗色适合夜间短操作，亮色适合长时间整理，两套主题信息层级完全等价。
- 色彩策略为 Restrained。海报承担主要色彩，品牌/操作色使用面积不超过 10%；所有颜色使用 OKLCH。
- 亮色核心：`bg oklch(1 0 0)`、`surface oklch(0.97 0.003 230)`、`ink oklch(0.20 0.015 230)`、`muted oklch(0.48 0.02 230)`、`border oklch(0.88 0.008 230)`。
- 暗色核心：`bg oklch(0.09 0 0)`、`surface oklch(0.14 0.006 230)`、`ink oklch(0.94 0.005 230)`、`muted oklch(0.68 0.012 230)`、`border oklch(0.28 0.008 230)`。
- 主操作色为 `oklch(0.54 0.15 230)`，白字对比度约 `4.66:1`；低频强调色为 `oklch(0.34 0.11 25)`。正文/次要文字对比度分别达到约 `18:1`/`6.5:1`（亮色）和 `17.4:1`/`7.2:1`（暗色）。
- 状态必须带图标与文字：想看（书签）、在看（播放/进度）、看过（勾选）、弃看（停止）；颜色仅作辅助。错误、警告、成功、信息使用独立语义色，不复用个人状态色。
- 字体使用自托管 Inter Variable + Noto Sans SC fallback，数字启用 tabular-nums；固定字号 `12/14/16/20/24/32px`，正文 `14/16px`，页面标题不超过 `32px`，字距统一 `0`。
- 间距以 4px 为基元，常用 `4/8/12/16/24/32/48px`；卡片圆角最大 8px，输入和按钮 6px，标签可用全圆角。阴影只用于浮层且模糊不超过 8px，不与装饰性边框叠加。
- 海报固定 `2:3`，网格使用稳定最小宽度；标题最多两行，年份与状态固定一行。缺图使用中性占位和片名字首，不绘制假海报。
- 桌面 `>=1280px`：216px 侧栏、64px 顶栏、最大 1440px 内容区；平板 `768–1279px`：72px 图标侧栏；移动 `<768px`：56px 顶栏、64px 底部导航，内容底部预留安全区。
- 影库海报网格：桌面最小 160px，平板 144px，移动双列且单项不小于 136px；固定比例避免加载和状态变化造成布局跳动。
- 详情桌面采用 `280px 海报 + minmax(0, 1fr)`，不放进装饰卡；移动端海报宽 132px 与标题区并排，主操作固定于安全区上方。
- 日历桌面显示完整月网格，移动切换为按日议程列表；统计页使用无卡片分区和可访问图表，不使用“巨大数字 + 小标签”的仪表盘模板。
- 所有组件必须定义 default/hover/focus/active/disabled/loading/error；加载使用保持尺寸的 skeleton，空状态说明下一步并提供单个主动作。
- 动效只表达状态，常规 `150–200ms`、抽屉 `220ms`，使用 ease-out；不做页面入场编排、弹跳或视差。`prefers-reduced-motion` 下移除位移和连续动画。
- z-index 语义层级固定为 dropdown 20、sticky 30、backdrop 40、modal 50、toast 60、tooltip 70。

### 第五章：技术架构、运维与发布（已确认）

- 仓库采用 `cmd/server` + `internal/{auth,media,records,household,integrations,sync,stats,storage,httpapi}` + `web` 的模块化单体结构；领域层不依赖 HTTP、TMDB 或具体数据库实现。
- Go 使用标准 `net/http` 兼容路由或轻量 `chi`，SQLite 使用无 CGO 驱动，SQL 迁移嵌入二进制；查询通过显式 repository 接口和类型化 SQL，不引入重量 ORM。
- React/Vite 前端使用 TypeScript strict、React Router、TanStack Query、React Hook Form + schema 校验、Radix primitives、Lucide；不引入全套视觉组件库，视觉令牌由项目 CSS 管理。
- API 固定前缀 `/api/v1`，JSON 使用 camelCase；分页使用游标；错误遵循 RFC 9457 Problem Details，附稳定错误码与 requestId。写操作使用 `Idempotency-Key`，可编辑聚合使用版本号/ETag。
- 认证使用数据库会话和随机不透明 cookie；cookie 为 HttpOnly、SameSite=Lax，生产可配置 Secure。密码采用 Argon2id，限制登录频率，会话可由管理员撤销。所有非 GET 同源请求校验 CSRF token 与 Origin。
- SQLite 启用 WAL、foreign_keys、busy_timeout；连接池限制单写、多读，所有迁移启动前执行，失败时 readiness 不通过且不继续启动业务端点。只允许单实例挂载同一数据卷。
- TMDB 搜索响应短缓存 6 小时；详情/季集快照 7 天，访问已记录条目时按 stale-while-revalidate 异步刷新。海报默认使用 TMDB CDN，不在首版长期镜像全部图片；缺图和 CDN 失败有占位。
- 媒体服务器适配器实现统一 `Provider` 接口；首版按实现顺序 Jellyfin、Emby、Plex。调度器读取持久化 job/sync_run 状态，默认 15 分钟增量同步、每日一次补偿同步；进程重启后补跑到期任务。
- 外部服务器凭据使用 `APP_ENCRYPTION_KEY` 派生密钥进行 AEAD 加密；数据库只保存密文、nonce、版本和指纹。环境变量密钥不进入备份；缺少或更换密钥时集成进入锁定状态，不损坏观影数据。
- `/healthz` 只证明进程存活；`/readyz` 检查迁移、SQLite 可写和调度器初始化，不把 TMDB/媒体服务器短暂故障作为不就绪。`/metrics` 首版不公开，设置页显示必要运行状态。
- 数据目录固定 `/data`，数据库 `/data/video-record.db`，临时备份 `/data/backups`。导出支持 JSON/CSV；系统备份使用 SQLite online backup API 生成一致性快照并附 manifest/checksum，不复制活动中的 WAL 文件。
- 恢复只能由管理员上传兼容备份；先校验版本、校验和与空间，再生成当前快照，进入维护模式后原子替换，失败自动回滚。升级前自动备份，备份保留策略可配置。
- Docker 多阶段构建：Node 构建前端，Go 关闭 CGO 构建静态二进制，最终镜像使用非 root 的精简 Debian/distroless 基础，包含 CA 证书与时区数据；前端嵌入二进制，容器只暴露 `8080`。
- 容器固定非 root UID/GID，`/data` 为唯一可写卷，根文件系统可只读，`no-new-privileges`、drop ALL capabilities；应用二进制提供 `healthcheck` 子命令，避免为健康检查安装 curl。
- 本机开发使用 `go run` + Vite dev server 代理 `/api`；生产 Compose 只有一个应用服务和一个命名卷，TMDB/加密/会话密钥从 `.env` 或 Docker secrets 注入，不带真实默认值。
- Git 只使用 `main`。每次提交必须通过 Go test/vet、前端 lint/typecheck/test/build、迁移测试、API 契约测试和 secret scan。
- GitHub Actions：push `main` 做验证；推送 `vX.Y.Z` 标签后用 Buildx/QEMU 构建并推送 Docker Hub 的 `linux/amd64,linux/arm64` manifest，生成 SBOM/provenance。标签为完整版本、主次版本和 `latest`；只有正式 semver 标签更新 `latest`。

## Task 16：一致性备份与原子恢复

- 备份使用 modernc SQLite Online Backup API 生成同一时点快照；数据库、manifest 与 SHA-256 均从快照文件流式生成，环境变量和加密密钥不会进入归档。
- 备份归档固定只含 `manifest.json` 与 `video-record.db`；列表只读取 manifest，下载使用普通文件流，目录内符号链接与路径型文件名被拒绝。
- 恢复上传流式写入 `/data/backups` 私有暂存文件并同步落盘，幂等哈希基于归档内容而非 multipart 边界；压缩与解压大小均有限制，不把生产数据库重复缓存在内存。
- 恢复在任何数据库读取前取得共享恢复锁；普通 API 请求进入活动计数，维护模式等待活动请求归零并返回稳定 `503 maintenance_mode`，`/healthz` 保持可用。
- 原子替换先对已 checkpoint 且关闭的旧数据库建立同文件系统硬链接，再用单次 rename 覆盖规范路径；新库重开、恢复审计提交和目录同步完成前保留旧链接，失败使用全新 30 秒恢复上下文原子回滚。
- manifest 的 schema version、迁移名称/校验和与快照数据库逐项一致后才允许恢复；真实 N-1 备份会在替换前迁移到当前版本，未来版本、未知迁移和篡改校验和被拒绝。
- 剩余空间按替换数据库、当前库在线快照、最坏归档开销计算；数据库与备份目录跨文件系统时分别探测，避免预恢复备份中途才暴露空间不足。
- `requiresEncryptionKey` 从完成后的快照读取；恢复缺少密钥时只返回 `integrations_locked` 警告，观影数据仍可访问。恢复审计不保存凭据或笔记，跨安装恢复时允许 actor 为空。
- 设置页备份/恢复区仅管理员可见；恢复使用确认对话框且初始焦点在确认按钮，错误以 alert 播报并保留已选文件。

## Task 18：Jellyfin 播放历史 Provider

- Jellyfin 核心 OpenAPI 不提供事件级播放历史；标准 UserData 只能表示条目当前/最近状态，无法可靠恢复重复播放。Task 18 因此使用官方 Playback Reporting 插件的管理员端点，并保留标准 `/Items/{id}` 作为元数据补全边界。
- 插件历史端点为 `/user_usage_stats/{userId}/{date}/GetItems`；查询类型必须使用数据库中的真实 enum 值 `Movie,Episode`。响应的稳定 `RowId` 作为分页/增量游标与重复播放事件身份。
- 插件响应只返回 `h:mm tt`，日期来自请求路径；适配器把二者组合为 UTC 事件时间，同时兼容完整 RFC3339 录制夹具。当前默认 `timezoneOffset=0`，与项目容器的 UTC 运行基线一致。
- 每个历史页按 `RowId` 升序、使用 `date|rowId` 游标；支持 `Since/Until` UTC 日期范围、默认 100/最大 200 条，并缓存同页重复条目的 Item 详情请求。
- 标准 `/Items/{id}` 的类型化响应只保留本地需要的 ProviderIds、标题、年份、季集和 runtime；电影/单集映射到统一 `ItemIdentity`，未知字段不会进入后续同步候选。
- 删除用户在认证检查中映射为稳定认证错误；401/403、429/Retry-After、5xx、超时、畸形/null JSON、非法 RowId、负 runtime、类型或条目不匹配均有显式边界。
- 上游正文和令牌不进入 Provider 错误；共享 conformance suite 验证认证、分页稳定性、取消、重试分类和脱敏。
- 全仓 race 暴露调度取消期间 SQLite 可能返回 `ErrTxDone`/`interrupted`；调度器现以父 context 为正常关停真值，不把取消竞争误报为持久化故障。

## Task 19：Emby 播放历史 Provider

- Emby 核心用户条目 API同样不能提供每次重复播放事实；事件级历史来自 Emby Playback Reporting 插件的管理员端点，通常位于服务器 `/emby` 基路径下。
- Emby 插件与 Jellyfin 版本协议不同：`GetItems` 返回包含 `user_id` 和 `activity` 的对象，时间为服务器本地 `HH:mm`，并可能附带 `RemoteAddress`；适配器使用专用 typed DTO，只保留同步需要的字段。
- 客户端接受可配置服务器 `time.Location`，按该位置选择历史日期并把本地插件时钟转换为 UTC；默认 UTC，避免把无时区字符串隐式交给进程本地时区。
- 历史请求使用 `/user_usage_stats/{userId}/{date}/GetItems?Filter=Movie,Episode`，标准条目详情使用 `/Items/{id}`；BaseURL 可显式包含 `/emby` 子路径。
- 稳定事件 ID 为 `emby:{rowId}`，游标为服务器本地 `date|rowId`；重复电影行保留为独立事件，同页重复条目只抓取一次 Item 详情。
- 历史 wrapper 的 `user_id` 必须与配置用户匹配，`activity:null`、畸形 JSON、非法游标、缺失 Item、删除用户、401/403、429、5xx 和超时均映射为稳定脱敏错误。
- Item DTO 会丢弃 Emby 媒体 `Path`，历史 DTO 会丢弃 `RemoteAddress`，这些服务器隐私字段不会进入候选、错误或日志。

## Task 20：Plex 播放历史 Provider

- Plex 核心服务直接提供事件级 `/status/sessions/history/all`，不需要额外插件；`historyKey` 唯一标识一次播放，`ratingKey` 稳定标识媒体条目，重复观看不会覆盖旧事件。
- 历史请求按 `X-Plex-Container-Start/Size` offset 分页，并固定 `accountID`、`sort=viewedAt:asc`、`includeGuids=1`；可选 `viewedAt>=/<=` 传入 UTC Unix 秒范围。
- Plex token 只通过 `X-Plex-Token` 请求头发送，不进入 query；客户端同时发送固定的产品、客户端 ID 和版本头，不包含用户数据。
- 适配器自动识别 XML/JSON MediaContainer，严格校验 `offset`、`size`、`totalSize` 与 Metadata 数量；null envelope、错误根节点、格式损坏和容器计数不一致均拒绝。
- Provider 事件 ID 为 `plex:{historyKey}`，Item Provider ID 为 `ratingKey`；TMDB/IMDb/TVDB 同时支持现代 `tmdb://` 等 GUID 与旧 `com.plexapp.agents.*://` scheme。
- typed DTO 不读取 JSON Media/Part 文件路径；认证、历史错误和同步候选不会携带 Plex 媒体文件系统信息。
- 重复 `historyKey` 通过统一 HistoryPage 校验拒绝，缺失 historyKey/ratingKey、非法时间、负时长/进度和未知媒体类型均不能进入后续候选。

## Task 21：候选匹配、冲突解决与同步界面

- 候选匹配优先使用当前账户已确认映射，其次使用 TMDB/IMDb/TVDB 外部身份；标题/原名与年份只能形成可能匹配，不能自动落档。
- 任一外部身份提示发生媒体类型冲突、多个外部 ID 指向不同条目，或目标已有手工/更高优先级状态时，候选必须进入冲突；手工 `completed` 不再成为自动落档例外。
- 自动应用只发生于精确且无冲突的候选；确认、重匹配与创建自定义条目在同一 SQLite 事务内写入确认映射、观看事件、参与者、剧集进度、个人状态/日期、候选状态与审计。
- 同步观看事件 ID 使用 `account_id:event_id` 命名空间；重复事件只复用原候选或映射，不重复创建观看事件，若试图把同一外部事实改指其他媒体则拒绝。
- 候选 payload 只保存统一 `HistoryEvent`、受控证据和匹配选项，不保存 Provider 原始响应、错误正文、媒体路径、基础 URL 或凭据；HTTP 响应再次收窄为前端实际需要的事件字段。
- Provider runner 在运行时按账户解密凭据，使用严格 JSON：Jellyfin/Emby 为 token + userId，Emby 可带 IANA timezone，Plex 为 token + accountId；未知字段、尾随 JSON和非法时区均返回不含秘密值的稳定错误。
- 增量任务延续持久化 Provider 游标，补偿任务固定回看 24 小时且不推进增量游标；每页再次执行统一 HistoryPage 校验，再交给候选服务幂等落档。
- 服务启动会为全部启用账户补齐增量/补偿任务，后台启动调度器，并把候选服务装配到受认证路由；凭据锁定或上游短暂失败不会阻塞 HTTP 服务。
- 同步状态 API 只返回当前用户账户的 provider/name/enabled、待核对数量和受控运行状态；候选查询与 confirm/rematch/ignore/custom 全部按当前用户隔离，写入复用 Origin、CSRF 与 Idempotency-Key。
- 设置页只显示紧凑账户状态与待核对入口，`/settings/sync` 使用非卡片列表展示图标+文字状态、匹配证据、单选候选和内联自定义表单；网络错误保留选择/输入，自定义展开后焦点进入标题。
- 浏览器发现同名同年候选的两个 radio 名称相同；修复后标签始终含候选序号，并在规范化原名与显示标题不同时展示原名，读屏与键盘用户可区分。
- `internal/sync` 最终语句覆盖率为 85.1%；真实浏览器在 `1440x900`、`768x1024`、`375x812` 无横向溢出，四种候选写入可用，控制台无 warning/error。

## Task 22：OpenAPI 合约与完整 HTTP 安全测试

- 路由与 OpenAPI 的可靠一致性检查需要比较已注册 method/path 集合，并强制每个 operation 有唯一 `operationId`、具体 schema 和实际响应媒体类型；仅检查 YAML 可解析不足以防止契约漂移。
- 写操作幂等不能采用“先查询、执行副作用、再插入”模式；两个并发请求会同时越过查询。SQLite 唯一键必须在副作用前原子预留，并明确区分 pending、completed、payload mismatch 和过期记录。
- 幂等重放的 Problem Details 必须保留第一次请求的 request ID，否则缓存正文与响应头会互相矛盾；空 `204` 响应也要保存非 null 空 BLOB，避免被误判为未完成。
- 恢复端点会原子替换数据库，因而不能在旧连接上完成通用幂等记录；恢复成功后的 finalize 必须显式使用重开的数据库，同时失败路径仍按普通 reservation 释放或缓存。
- 大文件上传不能套用通用 1 MiB 请求体缓存。导入可在 multipart handler 内按文件名和流式内容生成稳定哈希，恢复则按归档内容生成哈希，两者都保持原有大小和安全校验。
- 标签属于可编辑记录聚合的一部分；只给状态更新做 ETag 会留下丢失更新窗口。标签修改必须校验同一个 record version 并在事务中递增版本。
- API 文档中的 CSRF 不能只依赖全局文字说明；每个受保护写 operation 都需要显式 `X-CSRF-Token` header 参数，文件上传还必须声明真实 multipart 与下载 media type。
- 应用级 404/405 也属于 API 合约，应返回 `application/problem+json` 与稳定 code/requestId，而不是路由器默认纯文本。
