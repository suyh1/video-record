# 观看轮次、季级个人记录与单集时间设计

**日期：** 2026-07-14
**状态：** 已确认
**分支：** `main`

## 目标

把电影和剧集详情中的观看事实统一到明确的“观看轮次”中：电影按整部影片记录，剧集按季记录。每集直接展示并允许编辑精确到秒的观看时间；电影和剧集都移除原“观看历史”，改为可归档上一轮私人记录的“多刷”区域。

## 已确认行为

- 剧集个人状态、评分、私人笔记、观看方式和完成时间属于当前季，不再属于整部剧。
- 每集条目显示观看时间，精确到时分秒。
- 点击未看集前的圆圈，以当前时间标记已看。
- 已看集可以修改时间；时间不得晚于服务端当前时间。
- 未看集直接设置合法时间并确认后，同时变为已看。
- 电影和剧集详情均移除“观看历史”。
- 详情新增“多刷”区域和“再刷”按钮。
- 只有当前电影或当前季完成后才允许再刷。
- 再刷会完整归档上一轮，随后创建状态为“在看”的空白新轮次。
- 电影再刷清空当前轮次的完成时间、评分、笔记和观看方式。
- 剧集再刷只重置当前季的私人记录和逐集时间，其他季不变。
- 标签、片单、家庭分享设置和影视目录信息不随再刷重置。
- 项目处于调试阶段，不迁移旧个人记录或观看事实。

## 方案比较

### 方案一：显式、规范化的观看轮次

电影或剧集季拥有一个当前轮次和零到多个归档轮次。私人记录、分集观看时间和观看事件都关联轮次。该方案能够用数据库约束保证唯一当前轮次，归档内容可查询、可验证、可导入导出，确定采用。

### 方案二：当前记录加 JSON 快照

保留现有表，把再刷前的数据序列化到归档 JSON。实现较快，但字段校验、查询、统计、版本演进和逐集时间修改会依赖应用层解析，不采用。

### 方案三：从观看事件推导轮次

根据事件时间猜测第几刷，不新增轮次实体。事件删除、时间修改、同步补偿和跨日观看都会使边界不可靠，也无法准确关联当时的评分和笔记，不采用。

## 领域模型

### 观看轮次

新增 `watch_rounds`：

- `id`：本地 UUID。
- `user_id`、`media_id`：用户与影视作用域。
- `season_number`：电影为空，剧集为正整数季号。
- `round_number`：同一作用域内从 1 递增。
- `status`：`none`、`wishlist`、`watching`、`completed`、`dropped`。
- `rating`：可空的 0-100 整数。
- `note`：可空私人笔记。
- `viewing_method`：可空观看方式。
- `started_at`、`completed_at`：可空 UTC 时间。
- `archived_at`：为空表示当前轮次，非空表示只读历史轮次。
- `version`、字段来源与更新时间：继续支持乐观并发和来源优先级。

数据库使用局部唯一索引保证：

- 同一用户和电影最多一个当前轮次。
- 同一用户、剧集和季号最多一个当前轮次。
- 同一作用域的 `round_number` 唯一。

首次读取尚无轮次的作用域时返回版本 0 的空记录，不因 GET 写数据库。第一次保存、标记分集或完成电影时，在写事务内创建第 1 轮。

### 轮次分集记录

用 `round_episode_progress` 取代现有只支持当前事实的 `episode_progress`：

- 主键为 `round_id + episode_id`。
- 保存 `watched_at`、来源、关联观看事件 ID 和更新时间。
- 当前轮次和归档轮次使用同一结构；归档后只读。
- 只保存用户真正标记过的最小分集身份，继续按需使用 TMDB 分集目录。

观看事件增加 `round_id`。修改当前轮次的单集时间时，必须在同一 SQLite 事务内同时更新轮次分集记录和关联观看事件，保证详情、日历和统计一致。

### 影视级整理数据

新增 `user_media_profiles` 作为影视级投影和设置表，保存影库状态投影、标签/分享乐观版本、家庭分享开关与公开短评。标签、片单、TMDB 关联、家庭分享开关和影视目录继续以 `media_id` 为作用域，不进入观看轮次。多刷归档默认仅当前用户可见，不通过家庭共享暴露过去的私人评分或笔记。

## 状态规则

### 电影

- 当前轮次状态由私人记录表单控制。
- 保存为 `completed` 时必须提供精确到秒且不晚于当前时间的完成时间，并创建关联观看事件。
- 再刷归档已完成轮次并创建状态为 `watching` 的下一轮。

### 剧集季

- 季内第一集变为已看后，当前季轮次投影为 `watching`。
- 当前季常规分集全部已看后，投影为 `completed`，完成时间为该轮最晚的分集观看时间。
- 从完成季撤销任一集后，轮次回到 `watching`；全部撤销后回到 `none`。
- 再刷只归档当前季并创建该季下一轮，其他季的当前和历史轮次不变。

### 整剧投影

影库和首页仍需要单个整剧状态：

1. 任一季当前轮次为 `watching` 时显示 `watching`。
2. 否则使用最近更新季当前轮次的状态。
3. 首页下一集继续从当前轮次未看分集中选择，不读取归档轮次。

## 时间规则

- API 传输和数据库存储使用 UTC ISO 8601，并保持现有固定纳秒格式约束。
- 页面按浏览器本地时区展示 `YYYY-MM-DD HH:mm:ss`。
- 行内编辑使用原生 `datetime-local`、`step=1`，并动态设置本地当前时间为 `max`。
- 前端校验只用于即时反馈；服务端必须再次拒绝未来时间并返回 `invalid_watched_at`。
- 点击未看圆圈由客户端发送当前时间，服务端仍以自己的当前时钟执行上限校验。
- 电影完成时间升级为秒级日期时间，不再使用固定中午的日期占位。

## API 设计

新增或调整受会话保护的轮次接口：

- `GET /api/v1/records/{mediaID}/rounds/current?seasonNumber=N`
- `PUT /api/v1/records/{mediaID}/rounds/current?seasonNumber=N`
- `GET /api/v1/records/{mediaID}/rounds?seasonNumber=N`
- `GET /api/v1/records/{mediaID}/rounds/{roundID}`
- `POST /api/v1/records/{mediaID}/rounds/current/rewatch?seasonNumber=N`
- `GET /api/v1/records/{mediaID}/progress?seasonNumber=N`
- `POST /api/v1/records/{mediaID}/progress?seasonNumber=N`

电影请求不得传季号；剧集轮次和进度请求必须传正整数季号。服务端验证影视类型、当前用户、季号、轮次归属和归档状态。

所有更新继续使用：

- `ETag` / `If-Match` 乐观并发控制。
- `Idempotency-Key` 防止重复归档或重复创建事件。
- 现有会话、Origin 和 CSRF 保护。

稳定错误码包括：

- `invalid_round_scope`
- `invalid_watched_at`
- `round_not_completed`
- `round_archived`
- `version_conflict`

## 再刷事务

服务端在单个 SQLite 写事务中：

1. 锁定并读取当前作用域及版本。
2. 验证当前轮次存在且已完成。
3. 把当前轮次标记为归档，只读保留私人记录、完成时间和逐集时间。
4. 创建 `round_number + 1`、状态为 `watching` 的新当前轮次。
5. 保留旧观看事件及其轮次关联，不复制事件。
6. 提交后返回新当前轮次和更新后的历史摘要。

任一步失败都回滚；前端只有收到成功响应后才刷新私人记录、电影状态或当前季分集列表。

## 详情页结构

### 剧集

- 演员区域后显示作用于整个工作区的季选择器。
- 切换季时同时切换当前季分集、季级私人记录和该季多刷历史。
- 桌面保持现有双栏：当前季分集为主栏，季级私人记录为侧栏。
- 多刷区域位于双栏下方。
- 移动端顺序为季选择、当前季分集、季级私人记录、多刷。

### 电影

- 保留当前私人记录面板。
- 原观看历史位置替换为多刷区域。
- 电影完成时间使用秒级日期时间控件。

### 单集时间

- 已看集直接显示本地秒级时间。
- 未看集显示“设置观看时间”。
- 点击圆圈：未看立即记录当前时间；已看恢复未看。
- 点击时间或“设置观看时间”展开稳定高度的行内编辑区，提供确定和取消。
- 未看集确认合法时间后创建当前轮次分集记录并变为已看。
- 保存期间只禁用目标行和冲突动作，不让文字变化造成列表横跳。

### 多刷

- 标题行包含历史数量和带重复图标的“再刷”按钮。
- 当前轮次未完成时按钮禁用，并提供可访问说明。
- 历史行显示第几刷、完成时间和评分摘要。
- “查看”使用现有 Radix Dialog，展示完成时间、评分、私人笔记和观看方式。
- 剧集历史详情额外展示该轮各集的季集编号与秒级观看时间。
- 归档内容只读，不复用原观看事件删除界面。

## 数据清理迁移

本轮允许破坏性开发期迁移，不做旧语义兼容：

- 删除旧 `episode_progress`、`watch_events`、观看参与者、`user_media_states`、影视标签关系、共享评价和依赖观看事实的同步候选数据。
- 新建影视级 profile、观看轮次及轮次分集表，并为观看事件增加轮次约束。
- 保留用户、会话、影视目录、TMDB 身份、标签定义、片单及条目、集成账户和系统配置。
- 迁移测试必须证明清理范围准确，不能删除用户或影视目录。

## 同步、导入导出、日历与统计

- 手动、导入和已确认同步事件都必须进入一个明确轮次。
- 外部电影事件到达已完成当前轮次时，开始下一轮；剧集事件按季进入该季当前轮次。
- 日历和统计继续读取轮次关联观看事件，当前与归档轮次都参与统计。
- JSON 导入导出版本升级，结构化保存轮次、季号、私人记录和轮次分集时间。
- CSV 导出增加轮次号和季号列，不把归档私人记录公开给其他家庭成员。
- 备份恢复无需特殊分支，由嵌入式迁移和完整 SQLite 备份覆盖新表。

## 错误与并发处理

- 时间非法或位于未来：保留输入并在目标行显示错误。
- 版本冲突：重新读取当前轮次，保留未提交表单内容并允许显式重试。
- 再刷失败：页面不清空，不产生半归档轮次。
- TMDB 季目录失败：仍可读取已保存的季级私人记录和多刷历史；分集区域独立重试。
- 归档轮次写入：服务端返回 `round_archived`，不能通过旧标签页修改历史。
- 重复幂等请求：精确重放首次结果，不新增第二个轮次。

## 测试策略

### Go 与 SQLite

- 当前轮次唯一性、轮次号递增和电影/季作用域校验。
- 不同用户、不同季和电影记录隔离。
- 版本 0 首次写入、乐观冲突和幂等重放。
- 单集当前时间、手工时间、未来时间拒绝和事务内事件同步更新。
- 季状态投影、撤销、整季完成和再刷后其他季不变。
- 电影与季再刷原子性、失败回滚和归档只读。
- 影库状态、首页下一集、日历和统计使用正确轮次。
- 同步、JSON/CSV 导入导出和家庭隐私边界。
- 破坏性迁移清除旧记录但保留用户、影视目录、片单与配置。

### React

- 季切换同步更新分集、私人记录和多刷历史。
- 秒级展示、圆圈记录当前时间、行内编辑和取消。
- 未看集设置时间即变已看，未来时间保留并报错。
- 电影秒级完成时间和季级私人记录表单。
- 再刷成功后清空当前轮次，失败时保持原样。
- 多刷空态、禁用态、列表、摘要和只读详情弹窗。
- 键盘焦点、可访问名称、错误关联和稳定移动布局。

### 浏览器与全仓门禁

- 电影完成、再刷、归档查看和新轮次记录完整旅程。
- 单季再刷不影响其他季，归档逐集时间可查看。
- 375x812、768x1024、1440x900 的亮暗主题、无横向溢出和无控件遮挡。
- 秒级日期时间键盘操作、WCAG 2.2 AA axe 和 reduced-motion。
- Go race/vet/覆盖率、前端 lint/typecheck/test/API check/build、Playwright、视觉回归、依赖审计、补丁检查和凭据扫描。

## 不在本次范围

- 恢复或转换旧调试观看数据。
- 允许编辑归档轮次。
- 在家庭成员之间共享多刷私人笔记。
- 基于观看事件猜测或自动合并历史轮次。
- 改造 TMDB 演员、海报或季目录缓存策略。
